# Архитектура (для AI)

## Feature-Sliced Design

### Разрешённое направление зависимостей

Слой может импортировать **только из себя** и из слоёв **ниже** (ближе к `shared`):

```
app
  → pages
    → widgets
      → features
        → entities
          → shared
```

### Запрещённые импорты (примеры)

| Запрещено | Почему |
|-----------|--------|
| `entities/*` импортирует `features`, `widgets`, `pages`, `app` | домен не зависит от UI-сценариев |
| `shared/*` импортирует `entities` и выше | утилиты без домена |
| UI-компонент тянет бизнес-логику без прохода через store/hooks слоя | раздувание связности |

Исключения возможны только при явном решении владельца репозитория и с обновлением документации.

---

## Структура каталогов (сохранять)

Корень исходников:

```text
src/
  app/
  pages/
  widgets/
  features/
  entities/
  shared/
```

**Не вводить** новые корневые слои в `src/` (`services/`, `modules/`, `core/` и т.д.) без явного согласования.

Внутри слоя — сегменты в стиле проекта (`ui/`, `model/`, `api/` для `entities/chat`).

## Состояние и данные

### Zustand

- Единый стор мессенджера: `src/entities/chat/model/messenger-store.ts`.
- Новые действия и поля состояния, связанные с **чатами и сообщениями**, по возможности добавлять сюда (или в соседние типы `entities/chat`), а не создавать второй глобальный стор.

### Сид и сброс

- Начальные данные: `src/entities/chat/model/seed.ts` (и связанные типы `entities/user`, `entities/message`).
- Полное обнуление при перезагрузке страницы — ожидаемое поведение демо.

### Мок «бэкенда»

- `src/entities/chat/api/mock-backend.ts` — задержки и генерация текста ответа.
- Расширение мока (новые шаблоны, ветвления) — в этом модуле или рядом в `entities/chat/api/`, не в компонентах виджетов.

### Реальный API (на будущее)

- Тонкий слой вызовов рядом с сущностью или в `shared/lib` (HTTP-клиент, адаптеры).
- Компоненты **widgets/pages** не должны содержать длинную процедуру `fetch` + парсинг + обновление 10 полей стора в одном обработчике — вынести в функции/store actions.

## Поток «отправка сообщения» (ориентир)

1. Пользователь вводит текст в `features/send-message`.
2. Вызов action стора (например `sendText`).
3. Store добавляет исходящее сообщение, затем после `mockDelay` обновляет статус доставки, затем добавляет **ответ** другого участника через `buildMockReply` / логику выбора автора.

Любое изменение этого потока должно сохранять согласованность: порядок сообщений, `lastMessageAt`, `unreadCount` для неактивного чата.

## Запрещённые архитектурные изменения (по умолчанию)

Без явного запроса в задаче **нельзя**:

- переносить приложение на **Next.js, Remix, CRA** и т.п.;
- заменять **Zustand** на Redux, MobX, Recoil;
- ломать FSD ради «упрощения» (слить всё в `components/`, убрать слои);
- внедрять новый глобальный стейт-подход параллельно Zustand;
- подключать тяжёлый UI-kit или CSS-in-JS фреймворк только ради одной кнопки.

Допустимо: локальные `useState` в компонентах, небольшие утилиты, точечные новые файлы в правильном слое.

## Технологический контур

- Сборка: **Vite 8**.
- UI: **React 19**, **TypeScript 6**.
- Стили: **Tailwind CSS 4** (плагин `@tailwindcss/vite`).
- Алиас: `@/` → `src/` (`vite.config.ts`).

Изменение версий или замена инструментов — только по отдельной задаче и с проверкой `lint` / `build`.
