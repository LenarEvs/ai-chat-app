## Назначение проекта

**ai-chat-app** — одностраничный веб-мессенджер (UI в духе Telegram): список чатов, переписка, поиск, настройки чата и приложения, адаптив под узкие экраны.

- Данные **локальные**: Zustand-store + начальный сид; после перезагрузки страницы состояние сбрасывается.
- «Ответы собеседников» — **мок** (`delay` + шаблонные строки), не вызов LLM и не REST API.
- Цель репозитория — учебный/демо UI; бэкенд и реальный ИИ в коде **не подключены**.

Человеческий обзор для разработчиков: [README.md](../../README.md).

## Архитектура (кратко)

Проект следует **Feature-Sliced Design**: направление зависимостей **от `app` к `shared`**, без импортов «вверх» по слоям.

```
app → pages → widgets → features → entities → shared
```

- **Состояние мессенджера** сосредоточено в `entities/chat` (`useMessengerStore`).
- **Транспорт** сейчас — заглушки в `entities/chat/api/`; UI не должен содержать «сырую» имитацию сети рядом с разметкой без необходимости.

Подробно: [ARCHITECTURE.md](./ARCHITECTURE.md).

## Описание модулей (слои `src/`)

| Слой | Роль | Примеры в репозитории |
|------|------|------------------------|
| **app** | Корень приложения, проводка к странице | `App.tsx` → `MessengerPage` |
| **pages** | Композиция экрана из виджетов | `pages/messenger` — рейл, сайдбар, панель, настройки/профиль |
| **widgets** | Крупные самостоятельные блоки UI | `navigation-rail`, `chat-sidebar`, `chat-panel`, `settings-view`, `profile-view`, `chat-settings-view` |
| **features** | Пользовательские сценарии «одно действие» | `send-message` — форма отправки |
| **entities** | Домен: типы, сид, store, мок API | `chat` (store, seed, mock-backend), `message`, `user` |
| **shared** | Переиспользуемые утилиты и примитивы | `cn`, `use-media-query`, `Avatar`, `AppLogo` |

**Ключевые файлы логики**

- `src/entities/chat/model/messenger-store.ts` — чаты, сообщения, действия (отправка, поиск, настройки).
- `src/entities/chat/model/seed.ts` — начальные пользователи и чаты.
- `src/entities/chat/api/mock-backend.ts` — задержка и текст автоответов.

## Требования к ответам AI

### Формат ответа при изменении кода

1. **План** — короткие шаги до правок.
2. **Изменяемые файлы** — пути и тип операции (создание / правка / удаление).
3. **Объяснение изменений** — зачем так сделано; без лишней воды.
4. **Шаги проверки (validation)** — команды и сценарии из [DEFINITION_OF_DONE.md](./DEFINITION_OF_DONE.md).

Консультация без кода: можно сократить до плана и проверок идей.

### Шаги проверки (минимум)

1. `npm run lint` — нет новых ошибок в затронутом коде.
2. `npm run build` — если менялись типы, импорты, конфигурация.
3. Ручная проверка UI при изменении экранов/виджетов (`npm run dev`, сценарии из README).

### Тестирование

В репозитории **нет** обязательного набора автотестов (Jest/Vitest/Playwright не зафиксированы в `package.json`). Требования:

- не ломать существующие скрипты (`lint`, `build`);
- для багфиксов — воспроизведение сценария и явное описание «было / стало»;
- если добавляются тесты по запросу владельца — следовать выбранному в задаче инструменту и не вводить лишних зависимостей без согласования.
